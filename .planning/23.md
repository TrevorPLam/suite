### Appendix B: Full Drizzle Kit Configuration for Each Domain

The per‑domain migration strategy requires a separate Drizzle Kit configuration file for every bounded context. The `tablesFilter` is critical to prevent Drizzle Kit from generating destructive DROP statements for tables belonging to other domains. It scopes the migration generator to only tables with a specific prefix or exact name match.

**File: `packages/db/drizzle.calendar.config.ts`**

```typescript
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
  dialect: 'postgresql',
  schema: './src/schema/calendar/**/*.ts',
  out: './drizzle/calendar',
  dbCredentials: { url: process.env.DATABASE_URL! },
  schemaFilter: ['calendar'],
  tablesFilter: ['calendar_*', 'events', 'attendees', 'bookings', 'availability'],
  migrations: {
    table: '__drizzle_migrations_calendar',
    schema: 'drizzle',
  },
});
```

Repeat this pattern for each domain: drive (`drizzle.drive.config.ts`), vault (`drizzle.vault.config.ts`), mail (`drizzle.mail.config.ts`), etc. Each config sets its own `tablesFilter` and `migrations.table` name.

**Note on `tablesFilter` limitations:** This setting filters only tables, not sequences, extensions, or other database objects. If your domain uses custom sequences, you must also apply `schemaFilter: ['domain_name']` to restrict all operations to that PostgreSQL schema. The two work together to provide complete isolation.


### Appendix C: Sample Feature Specification (`.spec.md`)

Every feature begins with a spec file placed in `apps/<app>/specs/<feature-name>.spec.md`. The following is the required structure. The frontmatter is machine‑readable; subsequent sections are for human and AI consumption.

**File: `apps/calendar/specs/create-event.spec.md`**

```markdown
---
spec_version: 1
feature: create-event
app: calendar
status: draft
author: AI
created: 2026-06-04
---

## User Story
**As a** calendar user
**I want to** create a new event with title, start/end times, and optional description
**So that** I can schedule my day

## Acceptance Criteria
- [ ] AC‑01: Logged‑in user can access event creation form
- [ ] AC‑02: Form validates that end time is after start time
- [ ] AC‑03: Event is saved to database encrypted
- [ ] AC‑04: Event appears in user's calendar view after creation
- [ ] AC‑05: All data transmitted over HTTPS, payload encrypted before storage

## API Contract
- **Endpoint:** `POST /api/events`
- **Auth:** Required (session cookie)
- **Request Body:**
  ```json
  {
    "title": "string (required, min 1, max 200)",
    "startAt": "ISO date (required)",
    "endAt": "ISO date (required)",
    "description": "string (optional, max 2000)",
    "allDay": "boolean (default false)"
  }
  ```
- **Response (201 Created):**
  ```json
  {
    "id": "uuid",
    "title": "string",
    "startAt": "ISO date",
    "endAt": "ISO date"
  }
  ```

## Validation Rules
| Field | Rule | Error Code |
|-------|------|-------------|
| title | present, ≥1, ≤200 chars | `invalid_title` |
| startAt | valid ISO datetime | `invalid_start_time` |
| endAt | valid ISO datetime, > startAt | `invalid_end_time` |

## Error Cases
| Scenario | HTTP Status | Error Message |
|----------|-------------|---------------|
| Missing auth | 401 | `missing_session` |
| Invalid input format | 400 | `invalid_request` |
| Database write failure | 500 | `internal_error` |

## Database Changes
- Insert into `calendar.events` table (encrypted)
- Audit log entry in `app.audit_logs`

## Out of Scope
- Recurring events (separate feature)
- Notifications to attendees (separate feature)
- Event attachments (depends on Drive)

## Security Considerations
- E2EE: Event data encrypted before client‑side transmission
- RLS: Inserts scoped to current tenant
- Audit log: All writes recorded

## Performance Requirements
- Response time < 200ms for 95th percentile
- Concurrent creation: at least 100 events per second
```


### Appendix D: GitHub Actions Workflow Templates

#### D.1 CI Pipeline (`ci.yml`)

The CI workflow uses Nx affected detection to run only projects that changed and their dependents. It also includes the critical fix for caching the pnpm store.

**File: `.github/workflows/ci.yml`**

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - name: Get pnpm store directory
        id: pnpm-store
        run: echo "store_dir=$(pnpm store path --silent)" >> $GITHUB_OUTPUT

      - name: Cache pnpm store
        uses: actions/cache@v4
        with:
          path: ${{ steps.pnpm-store.outputs.store_dir }}
          key: pnpm-store-${{ runner.os }}-${{ hashFiles('**/pnpm-lock.yaml') }}
          restore-keys: pnpm-store-${{ runner.os }}-

      - run: pnpm install --frozen-lockfile

      - uses: nrwl/nx-set-shas@v4

      - run: pnpm nx affected --target=lint,typecheck,test,build

  compliance:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: anchore/sbom-action@v0
        with:
          path: .
          format: spdx-json
          output-file: sbom.spdx.json
      - uses: anchore/grype-action@v0
        with:
          sbom: sbom.spdx.json
          fail-build: true
          severity-cutoff: high
      - name: Generate VEX statements
        run: grype sbom.spdx.json -o vex --vex-status not_affected > vex.json
      - name: Upload VEX
        uses: actions/upload-artifact@v4
        with:
          name: vex
          path: vex.json
```

  push:

### Appendix F: Better Auth Rate Limiting Fix

**CVE‑2026‑45364** affects Better Auth versions prior to 1.5.0‑beta.11. The rate limiter keys IPv6 addresses individually based on the exact textual IP address from the `x‑forwarded‑for` header, without normalising the address or aggregating by `/64` prefix. An attacker controlling a `/64` allocation can rotate through 2⁶⁴ different source addresses to bypass rate limits. Versions 1.5.0‑beta.11 and later include the fix.

**Mitigation for the Sovereign Suite:**

- Pin `better-auth` to ≥1.6.2 in your `pnpm‑workspace.yaml` catalog.
- Apply rate limiting at Cloudflare’s edge level as a secondary layer. The `rate‑limit` Worker binding can inspect the entire IPv6 prefix by stripping the last 64 bits of the address before applying limits.
- Document in AGENTS.md that any custom rate‑limiting code must normalise IPv6 addresses to the `/64` prefix before keying.

### Appendix I: PNPM Catalog Best Practices

pnpm catalogs centralise dependency version management in the root `pnpm‑workspace.yaml`. The community strongly recommends **scoped catalogs** to group related dependencies (e.g., `catalog:frontend`, `catalog:backend`) and setting `catalogMode: strict` to block ad‑hoc versions outside the catalog.

**Example scoped catalog configuration:**

```yaml
# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
  - 'tooling/*'

catalog:
  frontend:
    react: ^19.0.0
    vite: ^7.0.0
    '@vitejs/plugin-react': ^4.3.0
  backend:
    hono: ^4.11.7
    drizzle-orm: ^1.0.0-rc.x
    pg: ^8.13.0

catalogMode: strict
```

**To enforce catalog usage across the monorepo**, integrate `pnpm‑catalog‑lint`, a Rust‑based linter that ensures every dependency in every `package.json` references the catalog. Without this discipline, developers may inadvertently pin versions directly, causing version drift across the 53 apps.

**Use `catalog:` in `package.json`:**

```json
{
  "dependencies": {
    "react": "catalog:frontend",
    "vite": "catalog:frontend",
    "hono": "catalog:backend"
  }
}
```


### Appendix J: Secret Scanning with Gitleaks and Pre‑commit Hook

Credentials must never be committed to the repository. The Sovereign Suite enforces secret scanning at two levels: a **local pre‑commit hook** for early detection and a **GitHub Actions workflow** for a final check in CI.

**Pre‑commit configuration:**

```bash
npx husky init
echo 'gitleaks git --pre-commit --staged' > .husky/pre-commit
```

**GitHub Actions step (`ci.yml` addition):**

```yaml
- name: Secret scanning with Gitleaks
  uses: gitleaks/gitleaks-action@v2
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Critical:** The CI gate catches secrets but the secret is already in the remote history at that point. The pre‑commit hook prevents this entirely.

**Additional recommendation:** Use TruffleHog in CI for broader detection of high‑entropy strings and rotated secrets. It scans all commits on every PR and can block merges when findings are detected. Combine Gitleaks for static pattern matching with TruffleHog for entropy‑based detection for complete coverage. The official GitHub secret scanner is also enabled for the repository and is mandatory for all private repositories. GitHub secret scanning push protection is the final gate.


### Appendix K: Mirroring Secrets into GitHub Actions with Doppler

Doppler replaces GitHub repository secrets as the single source of truth. All secrets except the Doppler token itself are fetched at runtime, never stored in GitHub.

**Step 1 — Add a Doppler service token to GitHub secrets as `DOPPLER_TOKEN`.**

**Step 2 — Use the Doppler fetch action in every workflow:**

```yaml
- name: Fetch Doppler secrets
  id: secrets
  uses: dopplerhq/secrets-fetch-action@v2
  with:
    doppler-token: ${{ secrets.DOPPLER_TOKEN }}

- name: Deploy Worker
  run: npx wrangler deploy
  env:
    DATABASE_URL: ${{ steps.secrets.outputs.DATABASE_URL }}
    BETTER_AUTH_SECRET: ${{ steps.secrets.outputs.BETTER_AUTH_SECRET }}
```

For improved security, use the OIDC (OpenID Connect) integration to avoid storing even the `DOPPLER_TOKEN` as a secret. Add `id‑token: write` permissions to the job and configure Doppler to trust the GitHub Actions OIDC provider. The Doppler secrets fetch action then authenticates via OIDC without any token in GitHub.

**Doppler sync to GitHub variables** is also supported: you can sync unmasked secrets as GitHub Actions variables for workflows that require environment‑wide configuration.


### Appendix L: Nx Cloud Self‑Healing CI Configuration

Nx Cloud Self‑Healing CI automatically fixes failing CI tasks by creating an agent, verifying the fix by re‑running the broken task, and pushing the proposed fix back to the PR for review.

**Step 1 — Connect your workspace to Nx Cloud:**

```bash
pnpm nx connect
```

**Step 2 — Enable Self‑Healing CI in the Nx Cloud dashboard** (Settings > Self‑Healing CI).

**Step 3 — Configure auto‑apply for verified fixes** (optional): Nx Cloud recommends tasks that look safe to auto‑apply based on your CI history. You can configure this in Settings > Self‑Healing CI > Auto‑apply verified code changes.

**Step 4 — Install the `ci-monitor` skill** (if using Claude Code or another agent):

```bash

```

This installs the `ci-monitor` skill, which enables the local agent to monitor pipeline progress in real time, receive failure information, and communicate with the Self‑Healing CI agent until the pipeline passes.

**End‑to‑end autonomous workflow:** Agent implements feature, pushes to CI, monitors status via `ci-monitor`, receives failure context, applies verified fix, repeats until all checks pass, and notifies human reviewer only when CI is green. This is the “Ralph loop” described in Section 20.

### Appendix N: Supplementary Notes on Third‑Party Tools

The following notes document evaluation results and known issues discovered during research for the Sovereign Suite.

#### N.1 Partyserver / PartyKit — Vite HMR WebSocket Bug

The Partyserver package (used by PartyKit) has a known issue where WebSocket handlers break after Hot Module Replacement (HMR) reloads when using Vite. The issue arises because the `createLazyConnection` property descriptor lacks the `configurable: true` flag, preventing proper redefinition during HMR. For the Sovereign Suite, this bug makes PartyKit unsuitable for development. The architecture instead uses raw Durable Objects (Section 10).

#### N.2 `tablesFilter` in Drizzle Kit

The `tablesFilter` option is now available in the `pushSchema` API as of PR #3141, merged in April 2026. However, a known limitation remains: `tablesFilter` only applies to tables. Sequences, extensions, and other database objects are not filtered. Using `schemaFilter` alongside `tablesFilter` is strongly recommended to achieve full isolation.

#### N.3 `create-mf-app` for Micro‑frontends

The `create-mf-app` CLI scaffolds production‑grade micro‑frontend platforms using React 19, Vite 7, and Module Federation. It supports both parent (shell) and child (remote) applications. The Sovereign Suite evaluated this tool but deferred Module Federation (Section 11.3.2) due to operational complexity. However, the tool remains an option for future self‑hosted enterprise deployments that require independently deployable UI modules.

#### N.4 Better Auth Email Verification

Better Auth natively supports email verification and password reset flows. The library handles email verification at its API endpoint directly without requiring a custom confirm page. The `verification` table stores one‑time tokens. The Sovereign Suite uses this built‑in flow rather than implementing a custom verification system. Version 1.6.11 fixes an edge case where email verification could auto‑sign‑in the wrong user when linking accounts.

#### N.5 Claud Code MCP Ecosystem

As of early 2026, the MCP (Model Context Protocol) ecosystem has matured significantly. Claude Code can now connect to MCP servers running inside a private network via a lightweight gateway that makes a single outbound connection, requiring no inbound firewall rules. The `claude-code-mcp` server extends Claude Code with tools for code explanation, review, fixing, editing, and testing, using the locally installed Claude Code binary.
